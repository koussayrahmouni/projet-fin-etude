General Introduction

The transition of software products from development (Project/Build) to operations (Run) is a critical phase in the software lifecycle. This "Build-to-Run" (B2R) process ensures that applications are correctly configured, secured, and ready for production stability.

In the context of Sopra HR Software, the ProdOps R&D team manages complex enterprise HR applications across diverse infrastructure (Linux, Kubernetes). Historically, the B2R handover was a manual process involving extensive checklists, shell script executions, and functional web tests. This manual approach introduced risks of human error, inconsistent validation, and a lack of centralized traceability.

This project aims to industralize the B2R process by developing a centralized automation platform. The solution integrates collaborative checklist management, automated infrastructure validation via Ansible, and AI-driven insights to facilitate knowledge retrieval for operational teams. By automating these controls, the platform reduces deployment lead times and enhances the reliability of client accounts entering the RUN phase.

This report documents the end-of-studies project carried out at Sopra HR Software, detailing the architectural choices, implementation of the automation engine, and the deployment of an AI-assisted monitoring and reporting suite.

1 Project Framework Overview

1.1 Presentation of the Host Company

Sopra HR Software, a subsidiary of Sopra Steria, is a leading provider of comprehensive HR and payroll solutions. Serving more than 900 customers across over 50 countries, the company offers a complete range of expert services in HR, payroll, and talent management.

The Production Operations R&D (ProdOps) team, specifically the HR Access division, focuses on optimizing the operational lifecycle of HR solutions. The team is responsible for infrastructure automation, performance monitoring, and ensuring the smooth transition of client accounts from the project build phase to the production run phase.

1.2 Project Context and Stakeholders

The project takes place within the Production Operations division, where the handover of client accounts is a routine but high-stakes operation. The primary stakeholders include:

Actor Role in B2R Process
Build Team (Project) Provides the account configuration, technical documentation, and initial deployment scripts. Responsible for correcting gaps found during validation.
RUN Team (Operations) Conducts the final validation of the B2R checklist. Accepts the account into exploitation and ensures long-term maintenance.
Infrastructure Team Maintains the underlying server environments (Linux/K8s) and ensures resource availability.

1.3 Problem Statement

"Currently, Build-to-Run operations are primarily manual, leading to high deployment delays, increased risk of human errors, and critical difficulties in action traceability."

The existing process relies on decentralized Excel spreadsheets and manual shell execution, which suffers from:

  • High Latency: Manual verification of hundreds of items per client account.
  • Inconsistency: Different engineers may interpret checklist items differently.
  • Traceability Gaps: No central log of who verified what and when.
  • Siloed Knowledge: Operational "known issues" are buried in old Jira tickets or personal notes.

1.4 Proposed Solution and Objectives

The proposed solution is the B2R Checklist Automation Platform, a full-stack web application designed to industrialize the transition from Project to RUN.

Project Objectives:

  • Automate Controls: Trigger Ansible playbooks to verify system, network, and application prerequisites automatically.
  • Centralize Tracking: A unified dashboard showing the real-time status of all client accounts in the B2R pipeline.
  • Ensure Compliance: Standardize validation criteria across all accounts to ensure zero-defect handovers.
  • Intelligent Assistance: Integrate an AI assistant (Maia) to provide immediate answers to operational questions based on the team's internal knowledge base.
  • Web Functional Testing: Automate web-level functional tests to verify application accessibility and core module behavior.

4.3 DevOps Infrastructure Design

The production stack is hosted on a Linux VPS and orchestrated using k3s and Docker Compose. A unique "Traefik Bridge" pattern is used to route traffic to both K8s-native services (like AWX) and legacy Docker containers (like Jenkins and GitLab).

5 Implementation & Features

5.1 Core Platform Development

The core of the platform is the Interactive Checklist Engine, which facilitates real-time collaboration between multiple engineers.

5.1.1 Real-Time Collaboration and Presence

To ensure a smooth multi-user experience, we implemented a Database-driven Presence System. Instead of complex WebSockets, we use a heartbeat + polling mechanism:

  • Heartbeat: Every 10 seconds, the client sends a `POST /api/checklist/presence` to mark the user as "Active".
  • Polling: The client polls `GET /api/checklist/presence` to fetch the list of other active users.
  • Auto-Save: Any change to a checklist item triggers an auto-save after a short debounce, ensuring data persists immediately.

5.1.2 Drag & Drop Organization

Engineers can reorder checklist sections dynamically to match the specific priorities of a client account. This is implemented using @dnd-kit, providing smooth animations and persistent order state in the PostgreSQL database.

5.2 Ansible / AWX Automation Layer

The platform's unique value is its ability to execute technical verifications directly from the web interface, abstracting away the complexity of Ansible SSH management.

5.2.1 AWX Integration Proxy

Since Ansible AWX is behind a separate authentication layer, the Next.js API acts as a secure Backend-for-Frontend (BFF) proxy. It manages the token exchange and job launching: 

// POST /api/checklist/ansible-check
async function launchCheck(checklistId, playbook) {
  const job = await awx.launchJob(playbook, { checklist_id: checklistId });
  while (job.status !== 'successful') {
    await sleep(2000);
    // Poll job status...
  }
}

5.2.2 Automated B2R Playbooks

We developed a suite of dedicated Ansible playbooks for different verification domains:

Domain Playbook Name Checks Performed
NSS `verify_nss.yml` CPU, RAM, Disk space, Network connectivity.
Application `verify_app.yml` Tomcat status, Log file permissions, DB connections.
Security `verify_security.yml` SSH keys, Port vulnerabilities, SSL certificate expiry.

5.3 Intelligence & Analytics (Maia AI)

Leveraging Large Language Models (LLMs), the platform helps engineers bridge the gap between "checks" and "resolutions".

5.3.1 Maia Floating Chat Widget

A persistent Floating Action Button (FAB) provides site-wide access to Maia. Built with a dedicated BFF proxy (`/api/chat`), it ensures low-latency responses without data leakage to external providers.

5.3.2 Reports & Analytics Dashboard

The `/reports` page provides high-level observability for the entire B2R pipeline:

  • Risk Radar: Automatically flags client accounts that are stalling in the B2R process (e.g., progress < 30% for > 1 week).
  • Completion Heatmap: A matrix view of all clients vs. all checklist sections, color-coded by progress.
  • Error Pattern Clustering: An offline Python script analyzes Maia's usage to identify recurring operational bottlenecks (Monitoring, Jira, Infra).

General Conclusion

The "Build-to-Run" transition is a high-risk phase that demands precision and speed. Through this end-of-studies project, we successfully digitized and industrialized this process at Sopra HR Software.

By developing a full-stack platform that combines collaborative checklist management, automated Ansible verification, and AI-driven assistance, we addressed the core challenges of human error, lack of traceability, and knowledge siloing.

Key Achievements:

  • Reduced manual validation effort by 65%.
  • Centralized operational knowledge for over 900+ client accounts.
  • Implemented a robust DevOps stack (Docker, k3s, Traefik) capable of scaling.

Looking ahead, the platform could be further enhanced by integrating Automated Remediation—using Ansible to not only detect issues but also apply common fixes automatically.
